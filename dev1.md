# 后端A (核心服务) 开发进度 - Dev1



本文档记录了后端核心服务（Backend-A, Dev1）的开发进度、关键说明和后续计划，旨在帮助团队其他成员快速理解当前系统状态并进行协作。



## 1. 当前进度



核心服务的骨架已经搭建完毕并经过了初步测试，目前已实现的功能符合`项目任务书.pdf`的MVP（最小可行产品）要求。

- **[已完成]** **WebSocket 通信网关**: 基于 Spring Boot 的 WebSocket 服务已启动并运行在 `/game` 路径。
- **[已完成]** **房间与会话管理**: `GameRoomManager` 可以动态创建和管理多个 `GameRoom` 实例。玩家连接后会被自动分配到房间，断开连接后会从房间移除。
- **[已完成]** **20Hz 服务器Tick**: `GameRoom` 内置了稳定的主循环，严格按照 **20次/秒** 的频率更新游戏状态并进行广播，符合项目性能指标。
- **[已完成]** **状态同步链路**: "客户端指令上报 -> 服务器处理 -> 游戏状态广播" 的核心数据链路已经完全打通。
- **[已完成]** **核心 DTO 定义**: `UserCommand`, `PlayerState`, `GameStateSnapshot` 等关键数据结构已定义完毕，构成了前后端通信的契约。

**结论**: 当前后端服务已经具备了作为一个权威服务器（Server-Authoritative）的基础能力，可以进行下一步游戏玩法的开发。



## 2. 如何运行与测试



其他开发者无需等待JavaFX客户端即可独立测试后端服务。

1. **运行后端**: 直接启动 `PixelStrikeBackEndApplication` Spring Boot 应用。
2. **使用Web测试台**:
   - 用浏览器打开项目根目录下的 `test-client.html` 文件。
   - 点击 **“连接”** 按钮，此时你应该能看到服务器开始以20Hz的频率持续发送 `GameStateSnapshot` 快照。
   - 在 **“发送指令”** 文本框中修改 `moveInput` 等参数，点击 **“发送指令”**。
   - 观察收到的快照中，对应玩家的 `x` 坐标等状态是否发生变化，以验证指令处理是否成功。
   - **强烈建议**: 打开多个浏览器窗口来模拟多玩家同房间的同步情况。



## 3. 给其他开发者的说明





### For Dev2 (业务服务)



- **房间分配**: 目前，为了方便测试，所有玩家都会被强制分配到一个名为 `"test_room_1"` 的房间。该逻辑位于 `GameWebSocketHandler.java` 的 `afterConnectionEstablished` 方法中。未来，这里的 `roomId` 需要由你开发的匹配服务或大厅逻辑来动态提供。



### For Dev3/Dev4 (客户端 A/B)



- **WebSocket Endpoint**: 服务器的WebSocket地址为 `ws://<服务器IP>:81/game`。本地测试时使用 `ws://localhost:81/game`。
- **通信协议 (DTOs)**:
  - **你需要发送的 (上行)**: `UserCommand.java`。请严格按照这个类的结构来构建你要发送的JSON对象。`playerId`由服务器在连接时分配，你无需填写。
  - **你会收到的 (下行)**: `GameStateSnapshot.java`。这是服务器广播的权威游戏状态，你的客户端渲染和逻辑应完全以此为准。
- **同步频率**: 你会以 **20Hz** (约每50ms一次) 的频率持续收到 `GameStateSnapshot`。客户端需要实现插值（Interpolation）算法来平滑地渲染两次快照之间的角色移动，避免卡顿。
